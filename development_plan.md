# LCJ Mall 次開発実装計画書

本計画書は、LCJ Mallのコードベース分析結果に基づき、将来の拡張性と保守性を向上させるための段階的な実装計画を提案するものです。

## 全体方針

- **守りのリファクタリングを優先**: 新機能開発の前に、まず技術的負債を返済し、開発の土台を安定させます。
- **段階的な改善**: 一度に大規模な変更を加えるのではなく、リスクを管理しながらフェーズを分けて進めます。
- **型安全性の維持**: tRPCとDrizzle ORMが提供する型安全性を最大限に活用し、デグレを防止します。

## 実装フェーズ

### Phase 1: 巨大ファイルの分割（最優先）

- **目的**: プロジェクト最大の技術的負債である `server/routers.ts` と `server/db.ts` を分割し、見通しを良くし、コンフリクトを減らし、将来の機能追加を容易にします。
- **期間**: 2〜3週間
- **タスク**:
  1. **ルーター分割 (`routers.ts`)**:
     - `server/routers/` ディレクトリを作成します。
     - 機能ドメインごと（例: `user.router.ts`, `product.router.ts`, `order.router.ts`, `receipt.router.ts` など）にファイルを分割します。
     - 各ファイルは自身のtRPCルーターを定義・エクスポートします。
     - `server/routers.ts` は、これらのサブルーターをインポートし、マージして `appRouter` を構築する責務のみを持つようにします。
  2. **DB関数分割 (`db.ts`)**:
     - `server/db/` ディレクトリを作成します。
     - こちらも機能ドメインごと（例: `user.db.ts`, `product.db.ts` など）にファイルを分割します。
     - 各ファイルは、関連するテーブルのCRUD操作関数のみをエクスポートするようにします。
     - `server/db.ts` は最終的に削除されるか、DB接続の初期化など最小限の役割のみを持つようにします。
- **注意点**: この作業は既存のコードの大部分に影響を与えます。一度に全てを分割するのではなく、例えば `user` 関連から着手するなど、少しずつ進めるのが安全です。各分割が完了するたびに、十分なテスト（特にリグレッションテスト）が必要です。

### Phase 2: 認証・セッション管理の整理

- **目的**: 複雑化した認証ロジックを整理し、保守性を向上させます。
- **期間**: 1週間
- **タスク**:
  1. **セッション情報の一元化**: `line_session` と `lcj_session` の2つのCookieを可能な限り統一することを検討します。例えば、LINEログイン成功後も `lcj_session` と同じ形式のJWTを発行し、単一のCookieで管理できないか調査します。
  2. **localStorageへの依存低減**: フロントエンドでの `localStorage` (`lcj_session_token`) へのフォールバック依存を減らし、可能な限りHTTP OnlyのCookieで完結するセキュアなセッション管理を目指します。
  3. **`me` エンドポイントの簡素化**: `lineLoginRouter.me` のユーザー特定ロジックを、セッションの一元化に合わせて簡素化します。

### Phase 3: レシート処理の堅牢化

- **目的**: OCR結果への依存度を下げ、レシート処理の信頼性とデータ整合性を向上させます。
- **期間**: 1週間
- **タスク**:
  1. **`lineReceipts` テーブルへのカラム追加**: `drizzle/schema.ts` を編集し、`lineReceipts` テーブルに `orderNumber` (varchar), `approvedOrderAmount` (int) などのカラムを追加します。
  2. **承認時データ保存ロジックの追加**: レシート承認時 (`receiptsRouter` の承認処理内) に、OCRテキストから抽出・確定した注文番号や金額を、新しく追加したカラムに保存するように `server/db.ts` (分割後は `receipt.db.ts`) の更新関数を修正します。
  3. **重複チェックロジックの変更**: 将来的に、重複チェックは `ocrRawText` の全文検索ではなく、この新しい `orderNumber` カラムに対して行うように変更します。

### Phase 4: 将来拡張に向けた準備

- **目的**: Phase 1〜3で安定化したコードベースの上に、将来の拡張機能（ミッション、独自CPSなど）の基盤を構築します。
- **期間**: 2週間〜
- **タスク**:
  1. **「解放可能ポイント / ミッション構造」の実装**:
     - `drizzle/schema.ts` に `userMissions`, `missionProgress`, `unlockablePoints` などの新しいテーブルスキーマを定義します。
     - Phase 1で分割された `server/routers/` と `server/db/` に、ミッションと解放可能ポイントを管理するための新しいルーター (`mission.router.ts`) とDB関数 (`mission.db.ts`) を作成します。
     - フロントエンドにミッション一覧画面 (`client/src/pages/Missions.tsx`) を作成します。
  2. **「独自CPS / ASP構造」の設計**:
     - 既存の紹介機能とは独立した、新しいアフィリエイトシステムの詳細設計を行います。
     - 必要なテーブル（`affiliateLinks`, `affiliateClicks`, `affiliateConversions` など）と、それらを操作するAPIの仕様を定義します。
     - この段階ではまず設計を固めることを優先し、実装は次の大きな開発サイクルで行います。

## 結論

この計画は、まず守りを固め、開発の足場を強固にすることから始まります。特に **Phase 1の巨大ファイル分割は、プロジェクトの将来を左右する最も重要なステップ**です。このリファクタリングを成功させることができれば、その後の機能追加や仕様変更は、より迅速かつ安全に行えるようになります。
